# Looms in production

> *A broken thread must stop the loom. Every failure prompts construction of the loom that should have caught it.* — Vision 17, [`sakichi_100_visions.md`](sakichi_100_visions.md)

This page records concrete instances of "build the loom that should have caught it" outside SPA AI itself. These are real upstream PRs against external repositories where a missing halt led to a real defect, the absent loom was identified, and the loom was then installed and ratified into a governance rule.

The point of this page is not to advertise SPA AI. It is evidence that the doctrine in [`mission.md`](mission.md) and [`sakichi_100_visions.md`](sakichi_100_visions.md) is something the maintainer applies to their own work, not just a frame for SPA AI's roadmap.

## Loom #1 — Maintainer Reception Audit Gate (MRA-1)

**Codified as**: OPS-RULE-034 v1, ratified D-VGC163-1 (2026-04-25)
**Weaver classes served**: maintainer (the recipient whose template-stated norm was violated), first-contributor (the contributor whose PR is at risk of close-without-merge).
**Failure that prompted it**: an AI-assisted PR ([ros-navigation/navigation2#6104](https://github.com/ros-navigation/navigation2/pull/6104)) opened with a body that violated the target repo's explicit norm. The repo's `.github/PULL_REQUEST_TEMPLATE.md` literally says:

> *Was this PR description generated by AI software? — Out of respect for maintainers, AI for human-to-human communications are banned.*

The PR was opened anyway because the local pre-submit gate (PSG-1, OPS-RULE-021) covered code correctness but not maintainer-reception fit. The violation was caught only by a post-open audit swarm — too late; the maintainer would have seen it first.

**5-Whys** (terminating at mechanism, not blame, per Vision 18):

1. We opened a PR whose body the template explicitly bans. → Never read the target's `PULL_REQUEST_TEMPLATE.md` before drafting.
2. Why didn't we read it? → Drafting treated body as an internal artifact, not as outbound submission constrained by upstream norms.
3. Why? → No step in our chain pivots from "what we know" to "what the recipient expects."
4. Why? → PSG-1 covers code correctness, not reception fit.
5. Why? → PSG-1 was designed for the failure mode we'd seen (silent acceptance). This was a different failure mode (norm violation) — and the loom for it had not yet been built.

**The installed loom**: a pre-PR-open swarm that pulls the target's `PULL_REQUEST_TEMPLATE.md` + `CONTRIBUTING.md` and validates the planned body against every numbered field and stated norm. Deviations require an explicit human override. Implementation lives outside SPA AI; the rule's text is at OPS-RULE-034.

**First post-ratification application**: alongside this page, an embedded-hal dogfood was prepared (a `.pre-commit-config.yaml` proposed via `PreCommitFormatterRustLoom`). Per OPS-RULE-034, the proposed PR ran the audit swarm pre-open. The audit caught two real issues — both before any maintainer was asked to review:

- **A defect in SPA AI's own loom**: the rustfmt hook shipped with `entry: cargo fmt --` (no `--check`), which silently rewrites Rust files and exits 0 — a no-op as a halt. Fix landed same day at [`aki1770-del/spa-ai#5`](https://github.com/aki1770-del/spa-ai/pull/5), with a regression test that asserts the `--check` flag and rejects the false "staged .rs files" claim from the PR body template. The "absent loom inside SPA AI" — a test that verifies loom-generated configs actually halt — got installed for this defect class.
- **A cold-PR risk**: the swarm flagged that an unsolicited tooling PR to a governance-sensitive repo (`rust-embedded/embedded-hal`) carried close-without-merge risk per the swarm's analysis of the repo's recent closed PRs (#734, #662 — same pattern). Honored by opening an issue first ([`rust-embedded/embedded-hal#736`](https://github.com/rust-embedded/embedded-hal/issues/736)) and waiting for maintainer signal before any PR. This is the rust-embedded-specific application of the bow-before-entering spirit (the cousin of OPS-RULE-022 ack-gate).

Both outcomes are exactly what MRA-1 was conceived to surface, and both happened structurally before any maintainer absorbed the cost. The dogfood PR itself remains in flight pending the issue reply — but the loom *learned two things this morning*, which is what looms are for.

## Loom #2 — Schema Input-Space Enumeration Gate (SISE-1)

**Codified as**: OPS-RULE-035 v1, ratified D-VGC163-2 (2026-04-25)
**Weaver classes served**: maintainer (review burden of edge-case bug reports later), adopter (downstream user whose nested-namespace input would have hit the parser ambiguity), driver (downstream-of-the-app whose costmap-filter behavior would have been wrong on the unhandled input shape).
**Failure that prompted it**: the same PR (#6104) introduced a new YAML schema for a costmap filter plugin. An audit swarm ran against the planned implementation and flagged a parser ambiguity in how the schema split keys on the first dot. The bug didn't trigger in typical usage (ROS doesn't allow dots in node names), but the schema was *ambiguous-by-construction* — exactly the kind of design gap that bites a year later when somebody tries an edge case.

**5-Whys**:

1. Why does the parser split on the first dot when target-node names could contain dots? → We assumed flat top-level identifiers; the example in our implementation plan used a flat name.
2. Why didn't a test catch it? → All tests used a single flat target-node name. No test exercised an ambiguity case.
3. Why? → Test design followed the implementation's intent, not the schema's actual input space. We tested what the code does, not what the schema admits.
4. Why? → No explicit "user-facing schema edge-case" gate in our pipeline.
5. Why? → Test discipline was built around regression prevention, not input-space coverage. Vision 95 (effort-asymmetry as a thread-break signal) had not been applied to the schema-design step.

**The installed loom**: any contribution introducing a new user-facing schema (YAML, CLI, JSON, gRPC) must produce an enumeration table covering edge-case inputs before the test matrix is sized. Categories: special characters in keys, nested namespaces, empty/missing/null values, boundary numeric ranges, type mismatches. The test matrix must include at least one negative test per axis. Enforced by the same audit swarm as MRA-1.

**Concrete fix in the same PR**: the parser was rewritten to require an explicit `target_nodes` list and use longest-prefix matching, with a regression test (`LongestPrefixMatchForOverlappingTargetNodes`) that exercises the previously-ambiguous case.

## Loom #3 — Literature Drift Detector

**Codified as**: `LiteratureDriftDetectorLoom` (sakichi vision 14 — silent
failure is anti-Jidoka — at the calibration layer), shipped in spa-ai 0.5.0.
**Weaver classes served**: maintainer (reviews the audit + decides per row), auditor (consumes the citation + date-source + drift + verified-marker provenance trail), driver (downstream of the app whose thresholds are calibrated to cited literature; ultimate beneficiary of re-verification).

**Failure that prompted it**: an aki1770-del-owned downstream package
(`navigation_safety_core` 0.4.0) shipped per-driver-class threshold magnitudes
calibrated to inline literature citations — `arxiv 2410.06388`, `PubMed
16313881` (Konstantopoulos novice fog hazard-perception), `PubMed 22664714`
(novice-fog crash-rate elevation). The pre-publish gate (a literature audit
performed by hand at release time) verified each citation was current in
April 2026. But nothing in the toolchain stopped a citation from rotting
silently across the next 12-24 months. Cited papers get retracted, preprints
reach final form with revised numbers, standards bodies amend, superseding
studies appear. Without a re-scan loom, the audit promise decays into a
publish-time snapshot — and downstream consumers absorb the drift in the form
of alert thresholds calibrated to year-old evidence.

**5-Whys** (terminating at mechanism, not blame, per Vision 18):

1. Why might a year-old preprint citation calibrate this year's alerts? →
   Nothing scans citations for age after publish.
2. Why doesn't anything scan? → The pre-publish audit was scoped to "is this
   citation real today?" — not to "will this citation still be current next
   year?"
3. Why was the scope narrow? → Maintaining a recurring scan cadence is a
   discipline that needs a loom; there isn't one.
4. Why isn't there a loom? → Literature drift is a slow failure mode; the
   pre-publish audit caught the first instance, so the discipline never
   formalised.
5. Why does the slow shape matter? → Slow failure modes are exactly the V14
   silent-failure pattern. The gauge that breaks once at publish IS the
   audit; without re-scan, the gauge then sits broken for months without
   surfacing.

**The installed loom**: per-language regex over source comments (Python `#`,
Dart `///` `//`, Rust `///` `//!` `//`, C++ `//`) capturing citation tokens
(`cite:`, `per <Author year>`, `arxiv <ID>`, `PubMed <ID>`, `PMC<N>`, `DOI`,
`RFC`, `ISO`). Date resolved by priority: explicit bracketed date in citation
→ embedded date in token (arxiv `YYMM`) → companion `# verified: YYYY-MM-DD`
marker on next line → otherwise marked `LOOKUP-REQUIRED` (PMID/DOI; the loom
makes no network calls) or `UNKNOWN-DATE`. Default threshold 18 months
(configurable via `.spa-ai-citation-drift.toml`). Built-in allowlist for RFC
and ISO identifiers; maintainer can extend or use the `[perpetual]` token
inline. The patch IS the audit (`.spa-ai-citation-drift.md`); no source
rewrite, ever.

**Why 18 months default and not the original 12-month proposal**: medical
treatment-guideline currency lives at ~24 months; aviation human-factors
review cycles at 2-5 years; automotive psychophysics underlying SNGNav-class
calibrations is decade-stable on mechanism but cohort-effect-shifting on
5-10 year scale. 12 months over-warns against stable evidence — itself a V14
risk against the maintainer's attention. 24 months under-warns against
fast-moving cybersecurity and ML literature. 18 months splits it; the audit
row records the threshold that fired so the maintainer sees by how much, and
per-project config tightens or loosens at one TOML line.

**First post-ratification application**: validated against the
`navigation_safety_config.dart` excerpt (lines 46-80) carrying the three
real citations above; the loom surfaces all three correctly (arxiv via
embedded-in-token date-source; both PubMed entries as `LOOKUP-REQUIRED`).

## Loom #4 (candidate, held) — Verify-First Gate post-audit-swarm

**Status**: candidate; held for the next governance docket.
**Weaver classes served (proposed)**: maintainer (no false-positive fix PRs land in their queue), first-contributor (no wasted effort drafting a patch for a non-bug).
**Failure pattern that prompted it**: audit swarms produce a fraction of false-positive findings — agents on truncated context invent plausible-sounding bugs in code they couldn't fully see. On the day MRA-1 + SISE-1 were ratified, two of three swarm-flagged "real bugs" turned out to be false positives when verified against actual current code (the bug shape was real-sounding, but the code already had the missing safeguard).

**The candidate loom**: between any audit-swarm finding and a "fix" PR, every BLOCKER/HIGH item must be verified against the actual current source via `gh api` source-fetch + grep. Each finding gets `[VERIFIED REAL]` or `[VERIFIED FALSE POSITIVE]` before the fix proceeds.

**Why held**: not enough evidence yet — accumulating one more swarm's worth of false-positive data before ratifying. Vision 65 (sekishō-idai, accumulate small things to achieve great things) — a rule shipped on two data points is a rule shipped too eagerly. The principle is operating informally already; codification waits for one more swarm.

## Reading these alongside SPA AI

The pattern in all three:

1. A real defect or near-miss is observed.
2. The 5-Whys terminate at a missing mechanism (Vision 18) — *which loom was absent?*
3. The loom is built, installed, and codified into a rule.
4. The rule fires structurally on subsequent work, so the same mechanism cannot fail twice for the same reason.

This is exactly what SPA AI's looms (`PreCommitFormatterLoom`, `PreCommitFormatterRustLoom`, and the looms to come) do at a smaller scale: each one is a Jidoka halt installed once at the right station. The two scales — repository-level rules like MRA-1, and commit-time hooks like rustfmt — are the same pattern at different altitudes.

If you're reading SPA AI for the first time, this page is the answer to *"is the maintainer actually doing this themselves?"* — yes, in the same week SPA AI shipped its first loom, two more looms were ratified in the maintainer's own governance and a third is queued. The doctrine is operational, not aspirational.
